## Context

当前仓库已积累大量与小说项目相关的脚本、监控、评审、人物档案与交付文档，但目录分布与命名不统一，造成以下问题：

- 团队协作链路断裂：上游产物缺少标准模板与签收记录，下游经常无法直接接续。
- 质量门禁缺失：有评审文档但缺少统一“通过线 + 驳回线 + 重写回路”。
- 平台交付割裂：微信公众号与小红书缺少统一的“改写规则 + 输出格式 + 发布前检查”。
- OpenClaw 运行不稳定：已存在 AGENTS/HEARTBEAT/MEMORY 等文件，但职责边界、更新节奏与故障升级未标准化。
- 仓库可维护性下降：根目录文件过多，历史产出与当前执行文件混放，检索和归档成本高。

本次设计面向“流程系统化 + 质量工程化 + 交付产品化”，优先沉淀标准，再执行重构与迁移。

## Goals / Non-Goals

**Goals:**

- 建立统一交付流水线与阶段门禁，使每一阶段都有明确输入、动作、输出和验收标准。
- 建立严审机制（强制驳回重写），保证不达标内容不能进入下游。
- 建立跨团队可追溯交接协议（版本、签收、驳回原因、回传时限）。
- 建立仓库信息架构与归档策略，控制新增混乱并逐步治理历史文件。
- 建立 OpenClaw 运营手册，固化记忆更新、心跳节奏、异常处理与 issue 升级流程。
- 建立微信公众号与小红书双平台交付标准，输出可直接发布的成品包。

**Non-Goals:**

- 不在本次设计中重写全部历史小说正文。
- 不在本次设计中接入新的第三方发布 API 自动投递。
- 不在本次设计中变更 OpenClaw 核心源码或 Gateway 内部实现。
- 不把“创作风格偏好”硬编码为唯一模板，而是提供可配置边界。

## Decisions

### Decision 1: 采用“主流程 + 子流程 + 门禁”的三级 SOP 架构

- 方案 A：仅维护一份主流程文档。
- 方案 B：主流程定义阶段与责任，子流程定义团队动作，门禁定义审查标准。✅ 采用
- 选择理由：兼顾全局可读性与执行可操作性；便于在不改主流程的前提下迭代单团队 SOP。

### Decision 2: 质量控制采用“双阈值 + 强制驳回 + 限次重写”

- 方案 A：单一总分通过。
- 方案 B：总分通过且关键项（事实准确、人物立体、逻辑一致、平台合规）必须全部达标。✅ 采用
- 选择理由：防止“总分过线但关键风险项不合格”的假通过。

### Decision 3: 团队交接使用“交付包协议”而不是口头状态

- 交付包必须包含：正文/素材、检查清单、评审记录、版本号、变更说明、下一步指令。
- 下游在固定时限内签收或驳回，并记录驳回码（如 FACT-01、NARR-03、STYLE-02）。
- 选择理由：让协作可追溯、可统计、可复盘。

### Decision 4: 仓库治理采取“先建标准，再迁移，再冻结规则”

- 步骤 1：定义目录与命名规范。
- 步骤 2：按批迁移历史文件，保留索引映射。
- 步骤 3：通过 README 与检查清单约束新增文件落位。
- 选择理由：避免一次性大迁移导致不可控风险。

### Decision 5: OpenClaw 运维采用“人格文件职责分离 + 记忆分层更新”

- AGENTS.md：行为规则与边界。
- SOUL.md/IDENTITY.md：人格与沟通风格。
- USER.md：用户画像与偏好。
- MEMORY.md + memory/YYYY-MM-DD.md：长期与日记忆分层。
- HEARTBEAT.md：定时巡检动作。
- TOOLS.md：本地环境特定信息。
- 选择理由：降低遗忘与越权风险，提升执行一致性。

### Decision 6: 平台发布采用“同源双稿”策略

- 同一小说产出两个成品：微信公众号长文版、小红书笔记版。
- 先进行“内容抽象层”提取，再按平台规范二次编排，而不是简单截断复制。
- 选择理由：保证跨平台质量与调性，不牺牲阅读体验。

### Decision 7: 异常升级采用“先自修后 Issue”标准流程

- 先执行本地排查与修复尝试（含证据）。
- 连续失败或超时进入 GitHub issue，按模板提交影响、复现、日志、建议方案。
- 选择理由：减少噪音 issue，提升问题处理效率。

## Risks / Trade-offs

- [风险] SOP 过重导致执行变慢
  → Mitigation: 采用“必选步骤 + 可选增强”分层，关键门禁不降级，非关键步骤可模板化自动化。

- [风险] 历史文件迁移期间路径失效
  → Mitigation: 建立迁移映射表与软链接/索引文档，迁移分批进行并逐批验收。

- [风险] 严格驳回机制导致团队挫败和堆积
  → Mitigation: 提供驳回码与可执行修复建议，限制单轮返工范围，明确重写时限。

- [风险] 平台规则变化导致发布模板过时
  → Mitigation: HEARTBEAT 纳入“平台规则抽查”任务，按月更新发布规范。

- [风险] OpenClaw 仍可能遗漏上下文
  → Mitigation: 强制会话启动检查、日记忆更新、交付前 checklist 签核。

- [风险] 多团队标准不一致
  → Mitigation: 统一模板仓（文档模板 + 审查模板 + issue 模板）并版本化。

## Migration Plan

1. 建立新目录骨架与文档入口（README 信息架构）。
2. 增量编写 SOP 主文档、团队文档、质量审查文档、平台发布文档。
3. 试运行一个人物样例，完成全链路演练与审查复盘。
4. 批量迁移历史文档到新目录，生成迁移映射索引。
5. 冻结“新文件必须按规范落位”规则，并纳入 HEARTBEAT 巡检。
6. 若迁移中出现异常，按“先自修后 issue”机制升级。

Rollback Strategy:

- 迁移采用批次提交，任一批次可通过 Git 回滚至上一个稳定点。
- 所有重命名/移动操作先生成清单，确保可逆。

## Open Questions

- 是否需要将“小说质量评分”拆分为两档：内部线（可继续打磨）与外部线（可发布）？
- 是否要求在每次驳回后自动生成“最小修改清单”，避免全量重写？
- 是否需要为不同题材（历史、人物传记、社会纪实）定义不同评分权重？
- 是否在后续阶段引入自动化 lint（命名、目录、元数据完整性）来辅助文档治理？
